# Zotero × Codex 论文研究助手

这套工具把工作分成三部分：

- **Zotero** 保存文献条目、PDF、合集、标签和引用信息。
- **Codex** 负责搜索、核验、筛选、精读、比较和整理研究缺口。
- **私有研究仓库** 保存每个课题的进度和研究结果，关闭任务或更换窗口后仍能恢复。

日常研究不需要运行 Python、Git，也不需要编辑 YAML 或 JSON。你只需要在 Codex 中描述研究任务，并在需要时到 Zotero 导入一次 RIS、补齐 PDF。

## 你只需要这样说

```text
开始研究柔性量子点光电探测器
搜索近五年的综述和代表性论文
把第 2、5、7 篇导入 Zotero
我已经导入并添加 PDF，继续
精读这三篇核心论文
比较它们的方法并整理研究缺口
```

下面按第一次使用到完成研究的顺序说明每一步。

## 一、第一次使用：连接 Zotero

### 1. 设置 Better BibTeX 自动导出

只设置一次。以后 Zotero 文献变化时，`library.bib` 会自动更新。

1. 打开 Zotero。
2. 在左侧栏右键“我的文库”。如果只想让 Codex 读取某个合集，就右键该合集。
3. 选择“导出文库”或“导出合集”。
4. 导出格式选择 **Better BibTeX**。
5. 勾选 **Keep updated**。这是自动同步的关键选项。
6. 保存位置选择：

   ```text
   <本仓库目录>\bibliography\library.bib
   ```

7. 导出完成后，不要手动编辑 `library.bib`。

然后在 Codex 中说：

```text
检查 Zotero 联动是否正常，告诉我读取到多少条文献和多少个 PDF 附件。
```

Codex 会检查当前 `library.bib`，不会读取旧的 JSON 缓存。如果结果是 0 条，直接看本文“常见问题”中的“读取不到 Zotero 文献”。

### 2. 确认 Zotero 条目和 PDF

Codex 只能精读 Zotero 中实际存在的 PDF。每篇重要论文至少检查：

1. 标题、作者、年份、期刊或会议是否正确。
2. DOI 是否存在且没有被错误截断。
3. PDF 是否挂在正确的文献条目下面。
4. 双击 PDF 能否在 Zotero 中正常打开。

合法添加 PDF 的常用方式：

- 使用 Zotero Connector 从出版社、数据库或开放仓储保存条目和附件。
- 将已经合法取得的 PDF 拖到对应 Zotero 条目上。
- 右键条目，选择“添加附件”再添加文件副本。
- 右键条目使用“查找可用的 PDF”；能否找到取决于开放获取状态和你的机构权限。

不要把 PDF 复制到 `research/`。原始 PDF 始终由 Zotero 管理。

### 3. 私有备份

本机的 `research/` 已连接独立私有仓库 `jia662-collab/paperlab-data`。每完成一次搜索选择、筛选批次、单篇精读或综合报告，Codex 会自动创建检查点并尝试推送。

你不需要操作 Git。若网络不可用，状态会显示 `pending_push`；成果仍保存在本机，下次“继续”时会先重试备份。

## 二、开始一个新研究课题

### 1. 新建 Codex 任务

在 Codex Desktop 中打开本仓库，然后为新课题新建一个任务。**一个任务只研究一个课题**，不要在同一任务里混入无关课题。

建议第一句话一次说明目标和边界：

```text
开始研究：柔性量子点光电探测器
目标：了解近五年的材料体系、器件结构、关键性能和未解决问题。
范围：2021-2026 年，中英文，优先综述和有完整实验数据的论文。
排除：只讨论刚性器件、没有全文、无法核验来源的文章。
```

信息不全时，Codex 最多追问一个真正影响检索的问题，然后创建项目。项目编号类似：

```text
20260711_柔性量子点光电探测器
```

项目创建后会出现：

```text
research/projects/<项目编号>/PROJECT.md
research/projects/<项目编号>/STATUS.md
research/state/<项目编号>.json
research/events/<项目编号>.jsonl
```

- `PROJECT.md`：研究目标、范围和人工决策。
- `STATUS.md`：当前阶段和下一步，内容很短。
- `state/*.json`：机器使用的真实执行状态。
- `events/*.jsonl`：审计记录，平时不会整份载入对话。

Codex 回复末尾会显示：

```text
当前项目：<项目编号>；下一步：<动作>
```

看到这一行，就说明当前任务已经绑定到正确课题。

## 三、搜索论文

### 1. 给出可执行的搜索要求

不要只说“找点相关论文”。至少说明时间、论文类型和关注点。例如：

```text
搜索 2021-2026 年柔性量子点光电探测器论文。
先找 5 篇高质量综述，再找 10 篇代表性实验论文。
重点关注响应率、探测率、弯折稳定性、器件结构和量子点材料。
排除只有摘要、来源无法核验和主题明显不相关的结果。
```

Codex 每轮默认最多给出 15 篇，并逐条核验标题、作者、年份、DOI 或稳定链接。结果保存到：

```text
research/projects/<项目编号>/search.md
research/projects/<项目编号>/candidates.json
```

`search.md` 是给你看的编号清单；`candidates.json` 是生成 RIS 时使用的结构化数据。

### 2. 缩小或补充搜索

可以继续使用自然语言调整结果：

```text
只保留有完整实验数据的论文。
再补一轮，重点找弯折半径小于 10 mm 的器件。
把综述和实验论文分开列出。
排除第 3、8、11 篇，并说明原因。
```

编号默认指当前项目最新的 `search.md`。搜索结果更新过多次时，为避免选错，最好同时说标题关键词：

```text
选择第 5 篇，也就是标题含 Quantum Dot-Based Flexible Photodetectors 的论文。
```

## 四、把选中的论文保存到 Zotero

默认使用 Zotero Connector，而不是批量 RIS：

1. Codex 先同步 Zotero，跳过 DOI 或标题已经存在的论文。
2. Codex 在 Chrome 中逐篇打开已核验的正式论文页面。
3. Zotero Connector 一次保存父条目和可获取的 PDF。
4. 父条目的正式 `Title` 保持不变；`Short Title` 使用 `ResNet`、`BatchNorm` 等简称。
5. 已有父条目但没有 PDF 时，只在该条目上使用“查找全文”。
6. 不通过 Zotero Connector 保存裸 PDF 页面，否则 Zotero 可能创建第二个父条目。

Connector 无法识别页面时，先使用 DOI 添加。只有 Connector 和 DOI 都失败，Codex 才为该论文生成单篇兜底文件：

```text
research/projects/<项目编号>/zotero_import.ris
```

兜底时在 Zotero 选择“文件 > 导入”，且只导入这一条失败记录。

不使用非法下载站，不绕过登录、验证码或付费墙。

等待 Better BibTeX 自动更新几秒，然后回到 Codex 说：

```text
我已经导入并添加 PDF，继续。
```

Codex 会重新解析最新 `library.bib`，按 DOI 优先、标题其次匹配刚导入的论文和附件。

## 五、直接使用 Zotero 中已有的论文

已有论文不需要重新生成 RIS。直接说：

```text
从我的 Zotero 中筛选柔性量子点光电探测器相关论文，优先有 PDF 的条目。
```

或指定更精确的条件：

```text
从我的 Zotero 中筛选 2022 年以后、标题或摘要涉及 flexible photodetector 的论文。
列出 citation key、标题、年份、DOI 和 PDF 是否存在。
```

Codex 会读取 Better BibTeX 的最新导出索引，不会修改 Zotero 条目，也不会移动附件。

## 六、快速筛选

告诉 Codex 你的判断标准：

```text
筛选刚导入的论文。
核心标准：必须研究柔性器件，并报告响应率、探测率或弯折稳定性中的至少一项。
把结果分成核心、重要、背景和排除四类。
每篇说明判断依据来自摘要还是全文，并列出仍需人工确认的内容。
```

筛选结果保存到：

```text
research/projects/<项目编号>/screening.md
```

如果没有 PDF，Codex 只能基于元数据或摘要筛选，并会明确标记“未使用全文”。摘要结论不会被当作全文证据。

你可以人工纠正：

```text
第 4 篇改为背景文献，因为它只讨论刚性器件。
第 6 篇保留为方法参考，不作为核心证据。
```

这些人工决定会写入项目记录，后续综合时继续生效。

## 七、精读一篇或多篇论文

### 1. 指定论文和阅读目标

你可以一次指定任意篇数。Codex 会在平台允许的并发数内同时处理，槽位不足时自动分批，不要求你拆任务：

```text
精读第 2、5、7、9、11、13 篇。
每篇都生成中文全文对照版和精读学习版。精读学习版必须沿原论文的论证顺序讲解，图表放在对应解释旁边。
```

也可以用标题或 citation key：

```text
精读 citation key 为 chauhanQuantumDotBasedFlexible2025 的论文。
```

### 2. Codex 实际如何读取

Codex 会从 Zotero 导出索引找到 PDF 路径，然后：

1. 计算源 PDF 哈希，不修改原文件。
2. 提取全文到本机忽略的缓存。
3. 先识别目录，再按章节完整读取，不设置固定页数或字符上限。
4. 检测到参考文献后停止；只有核验引用时才读取参考文献区。
5. 每完成一个自然章节就保存进度，长章节按自身小节继续，不遗漏正文。
6. 先用 PDF 布局信息自动定位 caption 和图表范围，再只渲染、检查最终裁剪；自动定位失败时才查看一次低清整页。
7. 需要核查具体数值、图表或争议时再返回对应原文页。
8. 同一次提取结果分别生成中文全文对照稿和精读学习稿，不重复解析整篇论文。

两份文稿的职责不同：

- **中文全文对照稿**忠实跟随原论文的章节、段落和页码，从摘要读到结论。它负责解决“原论文到底写了什么”。
- **精读学习稿**沿作者实际论证顺序解释上一节留下什么问题、本节如何回答、证据如何把论证推进到下一节。它负责解决“我应该怎样理解并学会这篇论文”。
- 重要图表在两份稿件中都保留原编号和原文 PDF 页码。翻译不确定、公式缺字或双栏错序会明确标记，不会猜测补写。

每篇论文保存三个可审计的研究文件：

```text
research/projects/<项目编号>/papers/<citation_key>.md
research/projects/<项目编号>/translations/<citation_key>.md
research/projects/<项目编号>/evidence/<citation_key>.json
```

- `translations/*.md`：按原文页码和章节排列的中文全文对照稿。
- `papers/*.md`：按作者论证顺序编写的精读学习稿。
- `evidence/*.json`：约 2,000 中文字符以内的证据卡，供多篇比较使用。
- `assets/<citation_key>/*.png`：仅保存在本机的论文图表，不上传 GitHub；缺失时可从 Zotero PDF 重建。

同时生成两份直接阅读的本地 PDF：

```text
output/pdf/<项目编号>/<短标题>_中文全文对照.pdf
output/pdf/<项目编号>/<短标题>_精读学习.pdf
```

这些 PDF 默认不上传 GitHub。旧 PDF 不会被静默覆盖；只有新文稿验收通过后才显式替换。

精读学习稿不再固定分成“一页读懂、问题、方法、结果、贡献”等栏目。目录优先沿用原论文目录，每一节用连续文字回答：作者为什么此时引入这一步，它解决上一节的什么不足，图表或实验怎样支持它，以及它为什么引出下一节。参数和复现步骤可以列表化，主要论证不能变成项目清单。

图表流程不会默认把 PDF 整页发送给 Codex：引擎先输出很短的布局 JSON 和建议裁剪坐标，Codex 通常只需要观察一次最终图片。这样既保留视觉核查，也避免反复查看整页造成的 token 消耗。

精读完成后，Codex 回复中会直接展示最重要的 1-2 张图，并给出两份 PDF 的绝对路径。需要重新查看时可以说：

```text
打开这篇论文的中文全文对照 PDF 和精读学习 PDF。
```

### 3. 实际怎样对照学习

1. 先打开 `<短标题>_精读学习.pdf`，看开头的阅读准备和作者论证路线，知道为什么要按这个顺序读。
2. 学习稿出现某个概念、数字、图表或公式时，查看旁边标注的原文 PDF 页码。
3. 打开 `<短标题>_中文全文对照.pdf` 的同一页码段，阅读该部分完整中文译文，确认学习稿没有省略关键前提。
4. 对重要结论仍有疑问时，在 Codex 中说：`核查精读学习版第 X 节的结论，回到原文 PDF 第 N 页解释。`

推荐在 Windows 中把两份 PDF 左右并排：左边放中文全文对照版，右边放精读学习版。你不需要在英文原 PDF、零散 Markdown 和证据 JSON 之间来回找内容；原 PDF 只在需要最终核查时打开。

### 4. 多篇论文如何并行精读

当你要求精读多篇论文时，Codex 使用一个协调代理和多个子代理：

1. 协调代理先锁定论文、citation key、原 PDF 路径和哈希。
2. 每篇论文分配一个子代理；每个子代理只读自己的论文，只写自己的译文、学习稿、证据卡和图片。
3. 共享的项目状态、综合报告、Git 提交和备份只由协调代理更新。
4. 如果你指定 6 篇而当前只能同时运行 4 个子代理，Codex 自动按 `4 + 2` 两波完成。
5. 某一篇失败不会取消其他论文；成功论文保留，只单独重试失败论文。
6. 每篇返回后先核对页码覆盖、图片和原 PDF 哈希，再生成正式 PDF；一批全部验收后统一备份。

你只需要说：

```text
并行精读这 6 篇。每篇生成中文全文对照 PDF 和按作者论证顺序编写的精读学习 PDF。
```

### 5. 特殊 PDF

- **扫描版**：Codex 标记 `scanned_suspect`，默认不做 OCR。需要时明确说“允许对这篇扫描 PDF 做 OCR”。
- **加密 PDF**：Codex 标记 `encrypted`，请在合法权限内换成可读取版本。
- **损坏 PDF**：Codex 标记 `corrupt`，请重新取得文件。
- **缺少 PDF**：Codex 只保留元数据判断，并给出合法全文查找路径。

## 八、多篇比较、争议和研究缺口

完成精读后直接下达综合任务：

```text
比较这三篇论文的材料、器件结构、样本条件、评价指标和主要结果。
统一单位，不能统一的指标分开列出。
区分真正的结论冲突和测试条件不同造成的差异。
最后整理证据充分的研究缺口和三个可执行研究问题。
```

Codex 每批最多读取 5 张证据卡，不会重新载入全部全文。只有具体数值或争议需要复核时，才读取对应 PDF 页。

综合结果保存到：

```text
research/projects/<项目编号>/synthesis.md
```

你可以继续要求：

```text
把比较结果改成适合写文献综述的结构。
只保留有页码或图表来源的结论。
把证据不足的研究缺口移到“待验证假设”。
根据现有证据设计一个最低可行实验。
```

## 九、如何“继续”和跨窗口恢复

### 原任务继续

在原 Codex 任务中说：

```text
继续。
```

Codex 会按以下顺序执行：

1. 读取当前项目的 `PROJECT.md` 和 `STATUS.md`。
2. 核对机器状态和实际成果文件。
3. 若备份是 `pending_push`，先重试推送。
4. 执行 `STATUS.md` 中记录的下一步。
5. 完成后保存新检查点。

“继续”不是重新分析全部论文，也不是依赖聊天记忆。

### 新窗口恢复

在同一仓库新建 Codex 任务，然后说：

```text
恢复项目：柔性量子点光电探测器
```

如果只有一个匹配项目，Codex 会恢复它；如果有多个相似项目，Codex 会列出名称让你选，不会按修改时间猜测。

恢复后先确认回复末尾的项目编号，再说“继续”。

### 同时研究多个课题

每个课题使用独立 Codex 任务：

```text
任务 A：研究｜柔性量子点探测器
任务 B：研究｜数字金融风险
任务 C：研究｜课程论文选题
```

不同课题写入不同项目目录和状态文件，不会共用一个全局“当前项目”。

## 十、结果文件在哪里

一个完成过搜索、导入、筛选和精读的项目大致如下：

```text
research/projects/20260711_柔性量子点光电探测器/
├─ PROJECT.md           研究目标、范围和人工决策
├─ STATUS.md            当前阶段和下一步
├─ search.md            编号后的搜索结果
├─ candidates.json      已核验候选论文数据
├─ zotero_import.ris    供 Zotero 手动导入
├─ screening.md         快速筛选结论
├─ papers/
│  └─ <citation_key>.md 逐篇详细笔记
├─ evidence/
│  └─ <citation_key>.json 供比较使用的短证据卡
└─ synthesis.md         比较、争议、缺口和研究问题
```

机器状态位于：

```text
research/state/<项目编号>.json
research/events/<项目编号>.jsonl
```

平时只看项目目录中的 Markdown 文件，不要手动修改状态 JSON 和事件日志。

## 十一、三个常用场景

### 场景 A：快速了解一个新方向

依次说：

```text
开始研究：固态电池界面稳定性。目标是两小时内建立领域地图。
搜索近三年的 5 篇综述和 10 篇代表性论文。
按材料体系、核心问题和主要方法筛选，不精读全文。
用筛选结果整理概念地图、主要争议和下一步阅读顺序。
```

这类任务可以先不导入全部候选，只把最终选择的论文导入 Zotero。

### 场景 B：精读 Zotero 中已经存在的论文

依次说：

```text
开始研究：我现有文献中的柔性光电探测器。
从我的 Zotero 中筛选相关论文，优先有 PDF 的条目。
先列出 citation key 和 PDF 状态。
精读其中最相关的三篇，重点提取器件结构和性能测试条件。
比较三篇结果并统一单位。
```

这个场景不需要生成或导入 RIS。

### 场景 C：研究中途加入一篇新论文

依次操作：

1. 用 Zotero Connector 保存新论文，或在 Zotero 中导入条目并补 PDF。
2. 等待 Better BibTeX 自动更新。
3. 在原 Codex 任务中说：

   ```text
   Zotero 中新增了一篇标题含 XXX 的论文。同步文献库，把它加入当前项目，先筛选再决定是否精读。
   ```

4. Codex 只处理新增论文，不重做已经完成的论文。

## 十二、完整示例：从空课题到研究缺口

### 第 1 步：建立课题

```text
开始研究：柔性量子点光电探测器。
关注 2021-2026 年的材料、器件结构、光电性能和机械稳定性。
```

预期结果：生成项目目录，下一步显示“搜索候选论文”。

### 第 2 步：搜索

```text
搜索 5 篇综述和 10 篇实验论文。必须有 DOI 或可核验的稳定链接。
```

预期结果：`search.md` 中出现编号后的 15 篇候选。

### 第 3 步：选择

```text
保留第 2、5、7、9 篇。第 2 篇作综述入口，其余作为实验对比。
```

预期结果：选择被写入项目状态并创建检查点。

### 第 4 步：通过 Connector 保存

```text
把第 2、5、7、9 篇导入 Zotero。
```

预期结果：Codex 跳过已有条目，从正式论文页面保存缺失条目；RIS 只处理 Connector 和 DOI 均失败的单篇记录。

### 第 5 步：在 Zotero 验收

1. 每篇论文只有一个父条目。
2. PDF 位于该父条目下面。

预期结果：四篇论文可并行处理；`translations/`、`papers/` 和 `evidence/` 中各有四个对应文件，`output/pdf/<项目编号>/` 中有八份 PDF。

### 第 8 步：综合

比较四篇论文的材料、结构、测试条件和关键性能。

预期结果：生成 `synthesis.md` 并保存私有备份检查点。

## 十三、常见问题

### 读取不到 Zotero 文献或显示 0 条

按顺序检查：

1. Zotero 中是否仍存在 Better BibTeX 自动导出。
2. 导出格式是否是 Better BibTeX。
3. 是否勾选 Keep updated。
4. 导出路径是否正好是 `<仓库目录>\bibliography\library.bib`。
5. 文件是否被导出到了另一个同名目录。

然后对 Codex 说：

```text
检查 bibliography/library.bib 的修改时间和条目数量，重新同步，不要读取旧缓存。
```

### Zotero 已有条目，但 Codex 找不到 PDF

常见原因：

- PDF 没有作为该条目的附件。
- 附件是网页快照或 HTML，不是 PDF。
- PDF 在另一台设备上，尚未通过 Zotero Sync 下载到本机。
- Better BibTeX 导出尚未刷新附件路径。

先在 Zotero 中双击附件验证，再说：

```text
重新同步这篇论文，按 DOI 匹配并检查 PDF 路径。
```

### 搜索结果能看到，但无法导入

如果 Codex 标记“未核验”，说明缺少可靠 DOI、作者信息或稳定来源。可以说：

```text
继续核验第 6 篇，只使用出版社、Crossref、PubMed、arXiv 或机构仓储等可靠来源。
```

核验成功后再生成 RIS。

### Connector 保存后没有 PDF

在已有父条目上使用 Zotero 的“查找全文”。不要从裸 PDF 页面再次点击 Connector；这会创建第二个父条目。仍无法获得时保留元数据，并记录为 `missing_pdf`。

### “继续”后项目不对

立即停止当前分析并说：

```text
不要继续。列出 research/INDEX.md 中的项目，让我选择，不要按修改时间猜。
```

选定后再说：

```text
恢复项目：<完整项目编号或名称>
```

### 备份显示 pending_push

这表示本机已经保存并提交，但暂时没有推送到私有 GitHub。研究成果没有丢失。网络恢复后说“继续”，Codex 会先重试备份。

### PDF 是扫描版、加密或损坏

先根据 Codex 返回的 `scanned_suspect`、`encrypted` 或 `corrupt` 处理原文件。OCR 默认关闭；只有明确允许后才会进行，并且 OCR 结果仍需人工核查。

## 十四、你不需要做的事情

- 不需要运行 PaperLab 命令。
- 不需要编辑 YAML、JSON 或 CSV。
- 不需要手动提交或推送 Git。
- 不需要复制、重命名或移动 Zotero PDF。
- 不要修改 `zotero.sqlite` 或 Zotero `storage`。
- 不要把 PDF、BibTeX、密钥或本机路径提交到公开仓库。
- 不要把摘要中的说法当作已经由全文证明的结论。

你真正需要做的只有三类决策：选择哪些论文、合法补齐哪些 PDF、确认或纠正哪些学术判断。

python -m pytest
